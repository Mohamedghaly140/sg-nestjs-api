## 2026-07-06 — Security · Enabled Row Level Security on all public tables

- The Supabase advisor flagged all 18 `public` tables as exposed with RLS disabled — since Supabase auto-exposes every `public` table through PostgREST as `anon`/`authenticated`, and this project's backend never uses those roles (it connects directly via `@prisma/adapter-pg` as the table-owning `postgres` role, which bypasses RLS), any leaked/used anon or service key would have granted full read/write access to every row.
- Added `prisma/migrations/20260706143950_enable_row_level_security`: `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` on all 17 app tables, deliberately with no policies (default-deny for every non-owner role).
- Applying this migration surfaced an unrelated pre-existing issue: `prisma migrate dev`'s shadow-database replay rejected the older `20260704234353_shipping_zone_null_city_unique` migration with a checksum mismatch (`P3006`) — its file's comment had been edited after that migration was already applied (see the 2026-07-06 "Prisma review hardening" entry below), leaving the stored checksum stale while the DDL itself was untouched (verified the live `shippingZones` partial unique index matches the file exactly). Fixed by updating the stored checksum to match the current file (user-approved, since it's a direct write to a shared database) rather than `prisma migrate reset`, which would have destroyed the seed data from the previous entry.
- **Codex review (`/codex:review`) caught two issues, both fixed:** (1) `_prisma_migrations`' RLS had initially been enabled via a manual one-off `ALTER TABLE` run directly against this Supabase project — not reproducible on a fresh database via `prisma migrate deploy`. Fixed with a follow-up migration, `prisma/migrations/20260706153859_enable_rls_prisma_migrations_table`, using `ALTER TABLE IF EXISTS` so it's a no-op on Prisma's shadow database (where the table doesn't exist) but takes effect on every real target database. (2) `docs/DATABASE.md` originally suggested adding RLS policies if the Supabase "anon/service" key were ever used client-side — wrong for the service-role key, which is designed to bypass RLS entirely and must never be exposed client-side regardless of policies. Reworded to separate anon (policy-governed) from service-role (never expose, full stop).
- Verified: `pnpm exec prisma migrate dev` applied both migrations cleanly; the app's own `postgres`-role connection is unaffected (re-ran `pnpm run seed` successfully after each migration); Supabase's security advisor no longer reports the critical "RLS disabled" finding (only informational "RLS enabled, no policy" notices, which is the intended default-deny state); `pnpm lint`/`test`/`test:e2e` all pass.
- Documented in `docs/DATABASE.md` (new §8 Database Security) and `docs/CODING_STANDARDS.md §6 Security`.

## 2026-07-05 — Phase 0 · fixes from Codex review of the PrismaModule work

- Added `"postinstall": "prisma generate"` to `package.json` — `generated/` is (correctly) gitignored, but nothing regenerated it on a fresh `pnpm install`, so a clean checkout couldn't build.
