## 2026-07-09 — Phase 11 Hardening · Checkout concurrency load test

- Added `test/load/checkout-load.ts` (`pnpm test:load`, autocannon-driven): seeds one low-stock product plus `CONCURRENCY` guest carts, fires exactly `CONCURRENCY` simultaneous `POST /orders/guest` requests at a running instance, and asserts the ADR-0003 conditional-stock-decrement invariant (exactly `STOCK` orders created, final stock 0) while reporting latency/throughput.
- Because this endpoint is throttled to 5/min/IP and a single load-test client necessarily shares one IP across all its simulated "users," added a `skipIf: () => process.env.LOAD_TEST_MODE === 'true'` escape hatch to the global `ThrottlerModule` config (`src/app.module.ts`) so the test can reach real DB-level concurrency. This is a rate-limit bypass mechanism living in shipped source — flagged by the auto-mode safety classifier during implementation, explained to the user, and proceeded with only after explicit sign-off. **Must never be set to `true` on a deployed instance** — documented in `CODING_STANDARDS.md` §6 and (once written) `docs/RUNBOOK.md`.
- **Real bug found and fixed:** the first run (`CONCURRENCY=10`) failed with all 10 requests getting a raw 500, not the expected `201`/`409` mix. Root cause: `lockAndValidateLines`' product-row lock fully serializes concurrent checkouts on the same product, and since `DATABASE_URL` is a remote Supabase-pooled Postgres, every query is a real network round trip — under 10-way contention, late-queued transactions could still be waiting on the lock past Prisma's default 5000ms interactive-transaction timeout, which kills the transaction and surfaces as 500 `INTERNAL_ERROR` instead of the correct 409 `INSUFFICIENT_STOCK`. Fixed by raising the checkout transaction's `timeout` to 15000ms (`CHECKOUT_TRANSACTION_OPTIONS` in `src/modules/orders/orders.service.ts`, applied to both `checkout` and `checkoutGuest` only — `claim`/`cancelMine` lock a single already-known order row, a lower-contention profile with no evidence they need it).
- Verified empirically against the real dev DB: 4/6/8-way contention all pass cleanly with the 15s timeout (max observed latency 4.1s/5.3s/6.8s respectively); 10-way contention is marginal (one run observed ~16.1s, exceeding the budget). Set the load test's default concurrency to 8 (reliably clean) rather than an arbitrary larger number, with `LOAD_TEST_CONCURRENCY` available to push further. Documented the residual limit (real fix is a reservation queue/Redis-backed hot-product locking, already future scope per `FEATURES.md` §12) as an addendum to `ADR-0003-stock-reservation-strategy.md`, which had explicitly called out "revisit if flash-sale patterns emerge."
- Docs touched: new `docs/testing/phase-11-load-test.md` (full write-up, methodology, results table), `ADR-0003-stock-reservation-strategy.md` addendum, `CODING_STANDARDS.md` §6, and this changelog. No schema/migration change.
- Verified: `pnpm lint`, `pnpm build`, `pnpm test` (47 suites / 226 tests), `pnpm test:e2e -- orders.e2e-spec.ts` (6 passed), and `pnpm test:load` (default concurrency 8) all pass against the real dev DB.

## 2026-07-09 — Phase 6 Checkout & Orders · Adversarial review hardening

- Fixed `OrdersService.claim()`'s read-then-write race: it looked up the order by `guestToken` (unlocked), locked the row, then unconditionally updated it by id — a second concurrent claim of the same still-valid token could pass the initial lookup before the first committed, then reassign `userId`/`claimedByUserId` after the first claimant had already consumed the token. Replaced the unconditional update with a conditional `updateMany` re-checking `{ id, guestToken: dto.token, guestTokenExpiresAt: { gt: now } }`; `count === 0` now collapses into the existing 404 `CLAIM_TOKEN_INVALID` path, matching the endpoint's documented no-oracle/idempotent-re-claim behavior in `docs/API_SPECIFICATION.md`.
- Fixed `CartService.loadCartForCheckout()` reading the cart without a row lock, while the cart was only locked much later (inside `clearCartInTx`, after order creation). Two concurrent `POST /orders` calls for the same authenticated user's cart (double-click, client retry) could both observe the same non-empty cart before either cleared it and, if stock covered both, both would commit a full order — a double-submit hole ADR-0003's stock-decrement guarantee doesn't cover on its own. `loadCartForCheckout` now locks the cart row immediately after finding it and re-reads the item list under that lock, so a second concurrent checkout blocks until the first commits/clears the cart, then observes an empty cart and hits the existing 422 `CART_EMPTY` path instead of duplicating the order.
- Fixed `AdminOrdersService.markPaid()` never checking order `status` — a CASH order already CANCELLED (stock restored, coupon released per FEATURES.md §6) could still be marked paid, incrementing `Product.sold` and breaking the revenue-reconciliation invariant in FEATURES.md §11 (`sold`/revenue excludes cancelled/refunded orders). Now also rejects `CANCELLED`/`REFUNDED` orders with the existing 409 `INVALID_STATUS_TRANSITION`. `API_SPECIFICATION.md`'s mark-paid error note updated to mention this case.
- Also fixed two wiring bugs surfaced while verifying the implementation against the real dev DB (not from the adversarial review): `CartIdentityMiddleware` was exported from `CartModule` without being registered in its `providers` array (Nest cannot export a non-provider); and `OrdersModule` applied that middleware to a raw string route path (`'orders/guest'`) which doesn't reliably match once the app's global prefix + URI versioning is layered on — both now follow `CartModule`'s own precedent of matching by controller class reference.
- All three adversarial-review findings were raised by a Codex adversarial review of the Phase 6 diff, independently reproduced by reading the code, and confirmed by the `fable-advisor` subagent (including narrowing the mark-paid finding to the CANCELLED case specifically, since REFUNDED is already unreachable while unpaid).
- Verified: `pnpm lint`, `pnpm build`, `pnpm test` (41 suites / 187 tests), and `pnpm test:e2e` (19 suites / 77 tests, including the must-pass concurrent-last-unit checkout race) all pass against the real dev DB. No schema/migration change.

## 2026-07-09 — Phase 5 Coupons & Shipping · Adversarial review hardening

- Fixed `CreateCouponDto`/`CreateShippingZoneDto` TS class-field initializers (`maxUsage = 0`, `perUserLimit = 1`, `isActive = true`) leaking through `PartialType`-derived update DTOs: `class-transformer` runs the initializers on `new UpdateXDto()` regardless of which keys the client sent, so a partial `PATCH` (e.g. `{ discount: 25 }`) silently reset the omitted fields and could reactivate a deactivated coupon or shipping zone. Fields are now optional with no initializer; create-time defaults come from the Prisma schema's own `@default(...)` (unchanged) instead of the DTO.
- Fixed `CouponsService.consumeCoupon` to lock the coupon row (`SELECT ... FOR UPDATE`, mirroring `CartService`'s `lockCart`/`lockProduct` pattern) and re-run the active/expiry/exhaustion/per-user-limit assertions under that lock before the atomic `usedCount` increment. Previously only the global counter was race-safe (via the conditional `updateMany`); the per-user limit was a separate, unserialized pre-flight read, so two concurrent checkouts for the same user against a `perUserLimit` coupon could both pass. `findEligibleCoupon` remains the lock-free preview/pre-flight check used by `POST /coupons/validate`; `consumeCoupon` is now the sole, fully self-validating commit point, matching the invariant already stated in `docs/DATABASE.md` §3.10.
- Fixed `AdminCouponsService.deleteCoupon`'s read-then-delete race (a concurrent checkout could consume the coupon between the existence/usage read and the delete, cascading its `CouponUsage` row and `SetNull`-ing the paid order's `couponId`) by replacing it with a conditional `deleteMany({ where: { id, usedCount: 0 } })`, falling back to `findUniqueOrThrow` only to distinguish "not found" (404 via `PrismaExceptionFilter`) from "in use" (409 `COUPON_IN_USE`) after a failed delete.
- All four fixes were raised by a Codex adversarial review of the Phase 5 diff and independently confirmed by the `fable-advisor` subagent, which also selected the row-lock approach for the per-user race over an alternative partial-unique-index design (rejected: a unique constraint can only encode `perUserLimit = 1`, not the documented arbitrary-integer/0-unlimited semantics).
- Tests added: unit coverage for the new lock-then-assert `consumeCoupon` per-user-limit branch and the conditional-delete branches; e2e regression coverage for partial-PATCH field preservation (coupons and shipping zones) and a same-user concurrent-consumption race (exactly one winner, one `COUPON_USER_LIMIT` rejection, `usedCount` lands at 1).
- Verified: `pnpm lint`, `pnpm build`, `pnpm test` (37 suites / 165 tests), and `pnpm test:e2e` (18 suites / 73 tests) all pass against the real dev DB. No schema/migration change; `docs/DATABASE.md` §3.10 already documented the invariant this fix now actually enforces.
